## Kimi tool-calling 問題排查結論

- 已排查 Kevin 回報的「Kimi 呼叫工具一直有問題」。
- 從 `~/.openclaw/agents/main/sessions/*.jsonl` 與 gateway log 確認：Kimi 在實際 tool-use 階段，會多次送出**空 arguments** 的工具呼叫，例如：
  - `exec {}`
  - `read {}`
- 對應錯誤明確包括：
  - `command: must have required property 'command'`
  - `Missing required parameter: path`
- 這表示問題不是單次意外，而是 **Kimi 在此 OpenClaw 環境中的 tool schema adherence 不穩定**，尤其在 `exec`、`read` 這類要求必填欄位的工具上會失手。
- 另外也發現 runtime / tool profile 有配置噪音：gateway log 反覆出現 `tools.profile (coding) allowlist contains unknown entries`，常見無效項包括：
  - `apply_patch`
  - `cron`
  - 有時還有 `image_generate`
- 結論：Kimi 的工具問題是 **模型穩定度不足 + tool profile 配置不乾淨** 兩邊一起造成。

## 對後續模型使用的實務建議

- Kimi 適合保留用於：
  - 中文寫稿
  - 腳本生成
  - 摘要整理
- Kimi **不適合當高工具依賴任務的主模型**，尤其是：
  - browser 操作
  - file ops
  - `exec`
  - config / debugging
  - 多步工具工作流
- 若任務核心在於穩定調工具，應優先改用更穩定的模型（如 GPT）作為 main / tool-heavy agent。
- 若仍保留 Kimi，至少應：
  1. 清掉 tool profile 裡不存在的 allowlist 項目
  2. 補強工具必填參數規則
  3. 接受它仍可能偶發空參數工具呼叫

## Kimi 工具呼叫修復結果（固化修復）

- 針對 Kimi 工具呼叫不穩問題，已採用 **docs-first + 本機配置修正** 方式處理。
- 實際修改的是正式配置檔：`~/.openclaw/openclaw.json`。
- 修復內容：將
  - `tools.profile: "coding"`
  改為
  - 明確 `tools.allow` 白名單
- 目的：避免 `coding` profile 帶入當前 runtime 不存在或不可用的工具名，減少 Kimi 被錯誤工具集/噪音誤導。
- 這次修復屬於 **固化修復**，不是臨時 prompt 或單次 session workaround；會隨 gateway 重啟與新 session 持續生效。
- Kevin 後續實測回報：Kimi 工具呼叫已恢復正常，表示本次配置修復有效。
- 實務結論：若之後再遇到 Kimi 工具問題，應優先檢查 `openclaw.json` 的工具暴露方式，而不是先假設單純模型抽風。
